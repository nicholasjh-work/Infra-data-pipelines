# Event Schema

This document outlines the events generated by the synthetic data pipeline.  The schema is intended to mirror the sort of logging used in a wearable health platform and downstream tools like Amplitude.  Each event captures a meaningful user action along with any associated attributes.

| Event Name | Description | Table | Notes |
|-----------|------------|-------|------|
| `feature_event` | A user engages with a specific feature in the app.  Fields: `member_id`, `event_date`, `feature` and `event_name` to indicate the type of interaction. | `RAW.FEATURE_EVENTS` | Used to compute weekly adoption metrics and track new versus repeat feature users. |
| `session_start` / `session_end` | The start or end of a user session.  Captures `member_id`, `session_start`, `session_end`, `device_type`, `os_version` and `location`. | `RAW.SESSIONS` | Session duration and device mix can inform user behaviour and segmentation. |
| `experiment_assignment` | When a member is enrolled in an experiment, this event records `member_id`, `experiment_id`, `variant` and `assignment_date`. | `RAW.EXPERIMENT_ASSIGNMENTS` | Downstream tables combine this with outcome metrics to evaluate A/B tests. |
| `subscription_update` | Records changes to a member’s subscription plan: `member_id`, `plan_type`, `start_date`, `end_date` and `auto_renew`. | `RAW.SUBSCRIPTIONS` | Helps measure churn and retention by plan type. |

### Amplitude considerations

The ingestion scripts in downstream repositories batch events before sending them to Amplitude.  The Amplitude HTTP API recommends sending no more than 10 events per batch and throttles requests above certain limits.  The loader therefore collects events into appropriately sized payloads and retries on HTTP errors.  Each event includes an `insert_id` to ensure idempotency.
